## 1. プロジェクト概要

NTT西日本が提供する「みんなのまちAI®」の中核機能を、**公表データのみ**で再現するオープンソースプロジェクト。  
フロントエンドに **Streamlit**、バックエンドに **FastAPI**、データストアに **DuckDB + Parquet** を採用し、**GitHub** で一元管理する。

### 1.1 解決する課題

| 課題 | アプローチ |
|------|-----------|
| 都市データが分散し、横断分析が困難 | 複数データソースを1プラットフォームで統合表示 |
| 施策検討にデータ根拠が不足 | シミュレーション＋LLMで根拠付き提案を生成 |
| 人流観測に高コスト（センサー設置等） | 重力モデル／Huffモデルで公表データから人流指数を推定 |

## 2. 機能要件

### 2.1 画面A — Explore（都市データ可視化）

**目的**: デジタル上にまちを再現し、多種多様なデータを地図上で重ね合わせ可視化する。

| # | 機能 | 詳細 |
|---|------|------|
| A-1 | マルチレイヤ地図表示 | 人口、年齢構成、産業、観光、公共交通、災害、POI、建物等のレイヤをON/OFF切替 |
| A-2 | 到達圏可視化 | 駅/バス停から徒歩N分の到達圏を等時間圏として表示（属性別：高齢女性等） |
| A-3 | 沿線人口×人流比較 | 鉄道沿線の世帯人口と駅周辺推定人流を同時表示 |
| A-4 | 時系列・自治体比較 | 指標の時系列推移グラフ、他自治体との比較パネル |
| A-5 | 統計カード | 地図クリックで当該メッシュ/地点の数値・出典リンクを表示 |

### 2.2 画面B — Scenario（シミュレーション）

**目的**: 地図上に施設・道路を描き、その介入による効果を定量的に可視化する。

| # | 機能 | 詳細 |
|---|------|------|
| B-1 | ポリゴン/ライン描画 | 地図上にドラッグで施設（ポリゴン）・道路（ライン）を描画 |
| B-2 | 介入パラメータ入力 | 用途カテゴリ（商業/公共/住宅等）、面積、階数、開業想定年、想定集客力 |
| B-3 | 到達圏差分表示 | 介入前後の等時間圏の差分をヒートマップで表示 |
| B-4 | 人流指数差分表示 | 重力モデルで計算した人流指数の変化を表示（ベース=100の相対指標） |
| B-5 | 恩恵人口表示 | 属性別（年齢、性別等）の恩恵人口を差分で表示 |
| B-6 | 推定注記の強制表示 | 全結果に「本指標は公表データに基づく推定値です」を強制表示 |

### 2.3 画面C — Budget Draft（予算案生成）

**目的**: プロンプト入力から根拠付き予算案を自動生成する。

| # | 機能 | 詳細 |
|---|------|------|
| C-1 | プロンプト入力 | 「〇〇に関する新規予算案を作成。対象=△△市、期間=R8-R10、上限=X億、目的=…」 |
| C-2 | 施策案タブ | A/B/C案（狙い、事業スキーム、概算費用、KPI、実施体制、リスク/対策） |
| C-3 | 根拠タブ | 根拠データ（APIジョブID・クエリ・集計定義）/ 根拠資料（公開計画・白書の該当章） |
| C-4 | 再現性タブ | 取得ジョブ一覧、更新ボタン、差分ログ |
| C-5 | 根拠拘束バリデーション | 各主張に根拠IDが1つ以上紐づかない場合は「仮説」枠に隔離し表示 |

## 3. 非機能要件

| 項目 | 要件 |
|------|------|
| レスポンス | Explore地図表示: 3秒以内、シミュレーション: 10秒以内 |
| 同時利用者 | MVP: 1〜3名（Streamlit制約） |
| データ鮮度 | 統計データは月次/年次更新対応 |
| 再現性 | 同一ジョブID→同一集計結果が再現可能 |
| 出典表示 | 全データに出典・ライセンス表示を必須化 |
| 推定注記 | 推定値には必ず「推定」ラベルを表示 |

## 4. データソース（公表データ縛り）

| データ種別 | ソース | 用途 |
|-----------|--------|------|
| 人口・世帯 | e-Stat（国勢調査メッシュ） | 人口分布、属性別分析 |
| 産業・経済 | e-Stat（経済センサス） | 就業者数、事業所数 |
| 建物・3D | PLATEAU (CityGML) | 建物ポリゴン、用途、階数 |
| 道路ネットワーク | OpenStreetMap (Overpass API) | ルート検索、到達圏計算 |
| 公共交通 | GTFS (各事業者公開分) | 交通ネットワーク・時刻表 |
| POI | OpenStreetMap | 商業・公共施設分布 |
| 行政界 | 国土数値情報 | 市区町村・町丁目境界 |
| 防災 | ハザードマップポータル / 国土数値情報 | 浸水想定区域、土砂災害 |
| 観光 | 観光庁統計 / 自治体公表資料 | 観光入込客数 |

## 5. 人流推定モデル設計

### 5.1 アプローチ: 重力モデル＋到達可能性の併用

```
人流指数(i→j) = k × P_i^α × A_j^β / d(i,j)^γ
```

- **P_i**: ゾーンiの発生量（人口、就業者数）
- **A_j**: ゾーンjの吸引量（POI数、床面積、魅力度スコア）
- **d(i,j)**: ゾーン間の移動抵抗（道路ネットワーク距離/時間）
- **α, β, γ**: パラメータ（デフォルト値設定、キャリブレーション可能）

### 5.2 出力形式

- **絶対値ではなく相対指数**（ベース=100）で出力
- 介入前後の**差分（Δ指数）**を可視化
- 全出力に「推定値」ラベルを強制付与

## 6. 受入基準

| # | 基準 | 検証方法 |
|---|------|---------|
| AC-1 | 地図で複数データを重ね、公共交通空白・沿線比較が再現できる | ExploreでA-1〜A-5の全機能動作確認 |
| AC-2 | 地図上で施設/道路を描き、到達圏と人流指数の差分が返る | ScenarioでB-1〜B-6の全機能動作確認 |
| AC-3 | プロンプトから予算案が出て、各主張に根拠が紐づく | Budget DraftでC-1〜C-5を動作確認 |
| AC-4 | 取得・集計・出力がGitHub上で再現可能 | CIテスト3種（再現性、根拠拘束、出典）がパス |

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| 人流指数 | 重力モデルで推定した相対的な人の流れの指標（ベース=100） |
| 到達圏 | ある地点から一定時間内に到達可能な範囲（等時間圏） |
| 恩恵人口 | 施策介入により到達圏改善の恩恵を受ける人口 |
| 根拠拘束 | 全主張に根拠ID紐づけを必須とするバリデーションルール |
| EBPM | Evidence Based Policy Making（根拠に基づく政策立案） |
